# AGENTS.md — AI Agent Instructions

## Project Overview

tree-sitter parser for OSGi Bundle MANIFEST.MF (OSGi R8 compliant).

## Design Document

See [`DESIGN.md`](DESIGN.md) for detailed implementation rationale and architecture.

## Key Files

| File | Editable | Description |
|------|----------|-------------|
| `grammar.js` | Yes | All rule definitions in tree-sitter DSL |
| `src/scanner.c` | Yes | External scanner (continuation lines, quoted strings, state management) |
| `src/parser.c` | **DO NOT EDIT** | Auto-generated by `tree-sitter generate` |
| `queries/highlights.scm` | Yes | Highlight queries |
| `queries/tags.scm` | Yes | Tag navigation queries |
| `test/corpus/manifest_test.txt` | Yes | Expected S-expressions for parser output (tests) |
| `DESIGN.md` | Yes | Design document |

## Development Workflow

1. Edit `grammar.js`
2. Edit `src/scanner.c` as needed
3. Add/update test cases in `test/corpus/manifest_test.txt`
4. Run `tree-sitter generate` to regenerate `src/parser.c`
5. Run `tree-sitter test` to ensure all tests pass

## Grammar Rules

### Coding Conventions

- Node names use OSGi R8 specification terminology (`clause`, `directive`, `typed_attribute`, `parameter`, etc.).
- Do NOT use the `osgi_` prefix — the parser's scope is self-contained.
- The `_` prefix denotes internal (private) rules (e.g., `_version`, `_version_interval`, `_unquoted_string`).
- When adding new rules, update the naming conventions section in `DESIGN.md` as well.

### Adding a Header

1. Add the following to the `choice()` in the `attribute_entry` rule:
   ```js
   seq(field('key', '<Header-Name>'), ': ', field('value', $.<rule_name>)),
   ```
2. Define the `<rule_name>` rule.
3. Add test cases to `test/corpus/manifest_test.txt`.
4. Run `tree-sitter generate && tree-sitter test`.

### Adding a Namespace Clause

When supporting a new namespace in `Require-Capability` / `Provide-Capability` (e.g., `osgi.cluster`):
- If possible, add a dedicated rule to the `choice()` in `capability`.
- If it conflicts with the tokenizer, fall back to the generic `clause` rule and handle namespace identification in queries (`highlights.scm`).

### Adding a Token

If a new token type is needed:
1. Add to `enum TokenType` in `src/scanner.c`
2. Add to the `externals` array in `grammar.js`
3. Implement the scan function in `src/scanner.c`
4. Add to the `valid_symbols` checks in `tree_sitter_..._external_scanner_scan()`
